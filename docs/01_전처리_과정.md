## 2. CSV 매칭 전략 — PDF·CSV 불일치(460 ≠ 456) 처리

`notebooks/src/data/loader.py` 의 `match_scorecard_to_directory()` 가 처리.

### 2.1 매칭 우선순위 (fallback 3단계)

```
1순위: project_id 정확 매칭        → 건물명 철자 달라도 100% 확실
2순위: project_name 소문자 비교     → id 실패 시 이름으로 재시도
3순위: 매칭 실패 → None             → PDF 데이터만으로 project 구성
```

### 2.2 매칭 실패 시 처리

CSV와 매칭 안 돼도 **분석에서 제외하지 않음**. `csv_match_node` 가 다음을 보장:

```python
# CSV 매칭 성공:
matched=True → cert_level, total_score, version 을 CSV 값으로 보완
              + GrossFloorArea (sq ft → sqm 변환: × 0.0929)

# CSV 매칭 실패:
matched=None → PDF 파싱값만으로 project dict 구성
              - 건물유형(ProjectTypes) = ""
              - 연면적(gross_area_sqm) = 0.0
              - 나머지는 PDF 값 사용
```

### 2.3 460 ≠ 456 원인 (가능성)

- CSV에는 없지만 scorecard만 확보된 건 (디렉토리 갱신 시차)
- 1개 프로젝트가 여러 scorecard PDF로 분리된 건 (예: IFC 1F + IFC B3, Alphadom 6-1 + 6-2)
- USGBC에서 confidential 처리된 프로젝트 (CSV 누락)

→ 불일치 건은 **PDF 데이터만으로도 표준화 가능하게** 설계됨. 결과적으로 `final_v5_data` 에는 460행이 채워짐.

## 3. PDF 파싱 (`pdf_ingest_node`)

### 3.1 추출 항목

`pdfplumber` 로 전체 텍스트를 뽑은 뒤 정규식으로 파싱.

| 필드 | 추출 방법 |
|------|----------|
| `version` | `(v4.1)`, `(v2009)` 등 괄호 패턴 |
| `leed_system` | `LEED O+M: ... (v4)` 형태 |
| `certification_level` | `PLATINUM/GOLD/SILVER/CERTIFIED` 키워드 |
| `project_id` (7자리 이상) | `^(\d{7,}),\s*(.+)$` |
| `total_awarded/possible` | `TOTAL 64 / 110` 패턴 |
| `categories` | `SUSTAINABLE SITES AWARDED: X / Y` 등 9개 패턴 |
| `credits` | `^(Prereq|Credit) <이름> X / Y$` 패턴 |

### 3.2 USGBC PDF의 "이중 문자" 버그 대응

일부 스코어카드에는 `SSUUSSTTAAIINNAABBLLEE` 처럼 문자가 2번씩 반복되는 인코딩 버그가 있음. `_fix_doubled_chars()` 가 두 단계로 복원:

1. 대문자 2쌍 이상 반복 → 1글자로 축약 (단, `EE` 1쌍은 보존해 `LEED` 유지)
2. 숫자·기호 1쌍 이상 반복 → 1글자로 축약 (`::` → `:`, `//` → `/`)

## 4. 버전 표준화 — 비율 환산 공식 (핵심)

### 4.1 수식

`rule_mapper_node` 가 사용하는 **결정론적 핵심 공식**:

```
                   awarded (구버전 획득점수)
score_v5(cat)  =  ─────────────────────────  ×  v5_max(cat)
                   old_max (구버전 만점)

단, 결과값을 v5_max 로 클램핑(상한 적용)하여 초과 방지.
```

**의미**: "이 카테고리에서 몇 %를 달성했는가" 라는 **달성률(achievement ratio)** 을 보존한 채, v5 만점 스케일로 점수를 재배정. 달성률이 변하지 않으므로 ML feature 로는 `ratio_{cat} = score_v5 / v5_max` 를 쓰면 버전 간 동일 기준으로 비교 가능.

### 4.2 수식이 내뱉는 것

소수점 2자리까지의 **v5 카테고리별 점수** (0 ~ v5_max(cat) 범위).

**예시** — v2.2 의 EA (17점 만점) 에서 12점 받은 건물을 v5 EA (33점 만점) 로 환산:

```
score_v5(EA) = (12 / 17) × 33 = 23.29   → 23.29점 (v5 EA 기준)
```

달성률 70.6% 가 그대로 보존됨 (`12/17 = 23.29/33 = 0.706`).

### 4.3 왜 "비율 환산"인가? — 절대점수 그대로 못 쓰는 이유

| 버전 | 총점 | EA 만점 | EA 비중 |
|------|------|---------|---------|
| v2.2 | 69 | 17 | 24.6% |
| v2009 | 110 | 35 | 31.8% |
| v4 | 110 | 33 | 30.0% |
| v5 | 100 | 33 | 33.0% |

버전마다 카테고리 총점도 총점 비중도 다름. v2.2 의 "EA 17점" 과 v4 의 "EA 17점" 은 **의미가 전혀 다름** (전자 100% 달성, 후자 52% 달성). 따라서 상대적 달성률 기준으로 환산해야 공정 비교 가능.

### 4.4 버전별 매핑 전략 상세

`VERSION_BD_C_MAX` 와 `rule_mapper_node` 의 분기 로직:

#### (A) v4.1 / v5 → 직접 매핑

구조가 완전 동일하므로 변환 없음. 카테고리 점수 그대로 사용.

```python
# v4.1 점수 = v5 점수 (이미 호환)
mapped["EA"] = float(cats.get("EA", 0))
mapped["LT"] = float(cats.get("LT", 0))
# ... 모든 카테고리 동일
```

#### (B) v4 → WE만 비율 환산

v4 와 v5 의 차이는 **WE 만점이 11점 → 12점** 으로 늘어난 것뿐. 나머지는 구조 동일.

```python
# WE 만 환산:
mapped["WE"] = (awarded_WE / 11) × 12    # v4 → v5

# 나머지는 직접 복사:
mapped["EA"] = awarded_EA
mapped["LT"] = awarded_LT
mapped["SS"] = awarded_SS
mapped["MR"] = awarded_MR
mapped["EQ"] = awarded_IEQ    # v5에서 IEQ → EQ 코드 변경
mapped["IP"] = awarded_IP     # v4부터 존재
```

#### (C) v2009 / v3 → 총점 110, SS에서 LT 분리 + 4개 카테고리 비율 환산

**구조 차이점**:
- v2009 에는 LT 카테고리가 **없음**. 교통 크레딧이 SS (26점) 안에 ~6pt 섞여있음.
- IP 카테고리 없음 (v4에서 신설).

**변환 로직**:
```python
# 1. SS 에서 교통 크레딧 분리 (아래 4.5절 상세)
ss_transport_awarded, ss_transport_max = <분리>
ss_pure_awarded = ss_awarded - transport_awarded

# 2. LT (신규 생성):
mapped["LT"] = (ss_transport_awarded / ss_transport_max) × 15   # v5 LT max

# 3. 순수 SS:
mapped["SS"] = (ss_pure_awarded / 20) × 11    # v2009 SS 26 - 교통 6 = 20

# 4. 비율 환산 (4개):
mapped["WE"] = (we_awarded / 10) × 9
mapped["EA"] = (ea_awarded / 35) × 33
mapped["MR"] = (mr_awarded / 14) × 18
mapped["EQ"] = (ieq_awarded / 15) × 13

# 5. IP 없음 → 0
mapped["IP"] = 0

# 6. IN, RP 는 v5 에서 폐지 → dropped_categories 에 기록만
dropped["IN"] = ieq_awarded  # 점수엔 반영 X
dropped["RP"] = rp_awarded
```

#### (D) v2.2 / v2.0 → 총점 69, SS에서 LT 분리 + 거의 전부 비율 환산

**구조 차이점**:
- 총점이 69점 (v2009 이후는 110점).
- LT, IP, RP 모두 **없음**.
- SS 14점 중 교통 ~7pt.

**변환 로직**:
```python
# 1. SS 에서 교통 분리
transport_awarded = <분리>

# 2. LT (신규):
mapped["LT"] = (transport_awarded / 7) × 15

# 3. 순수 SS:
mapped["SS"] = (ss_pure / 7) × 11   # v2.2 SS 14 - 교통 7 = 7

# 4. 나머지 전부 비율 환산:
mapped["WE"] = (we / 5)  × 9
mapped["EA"] = (ea / 17) × 33
mapped["MR"] = (mr / 13) × 18
mapped["EQ"] = (ieq / 15) × 13

# 5. IP, RP = 0 (카테고리 자체가 없음)
mapped["IP"] = 0

# 6. IN은 v5에서 폐지 → dropped 에 기록만
```

### 4.5 SS → LT 교통 크레딧 분리 (구버전에만 해당)

v2.2 / v2009 등 구버전은 LT 카테고리가 없고 교통 크레딧이 SS 안에 섞여있음. v5 의 LT 로 분리하는 2단계 fallback:

**우선 1: PDF 크레딧 상세 데이터 활용 (`credit-exact`)**

PDF 에 개별 크레딧이 파싱돼 있으면 키워드 매칭으로 교통 점수 정확히 추출:

```python
TRANSPORT_KEYWORDS = (
    "alternative transportation", "public transportation", "transit",
    "bicycle", "low-emitting", "fuel-efficient", "parking capacity",
    "green vehicles", "electric vehicle",
)

# 크레딧명에 위 키워드가 들어간 것의 awarded 합산 = transport_awarded
# 해당 크레딧들의 possible 합산 = transport_max
```

**우선 2: 역사적 평균 비율 추정 (`ratio-estimated`)**

크레딧 상세 데이터가 없으면 (v2009 의 114 건이 이에 해당) 버전별 평균 비율로 추정:

```python
SS_TRANSPORT_MAX = {
    "v2.2":   7,    # SS 14점 중 교통 크레딧 4.1~4.4 합계 = 7점
    "v2009":  6,    # SS 26점 중 교통 크레딧 4.1~4.4 합계 = 6점
}

# SS 획득점수 중 교통 비율만큼 비례 배분:
transport_ratio = SS_TRANSPORT_MAX[version] / SS_total_max
transport_awarded = SS_awarded × transport_ratio
ss_pure_awarded = SS_awarded - transport_awarded
```

⚠️ 이 추정은 **"교통 크레딧 달성률 ≈ SS 전체 달성률"** 가정에 의존함. 이 가정이 틀릴 경우 drift 가 커져 hallucination_check 에서 FAIL → LLM 구제 경로로 이동.

## 5. 카테고리별 "원래 만점(old_max)" 결정 우선순위

`rule_mapper_node` 의 `get_old_max(cat)` 함수가 구버전 카테고리 만점을 결정할 때 쓰는 3단계 fallback. 이 값이 분모(`old_max`)가 되기 때문에 정확해야 비율 환산이 맞음.

### 5.1 3단계 우선순위

```
1순위: PDF 의 categories_possible[cat]  (스코어카드에 적힌 possible)
2순위: rubric xlsx 조회                  (data/rubrics/{version}/*.xlsx)
3순위: VERSION_BD_C_MAX 하드코딩         (BD+C 기준 기본값)
```

### 5.2 왜 PDF possible 이 최우선인가?

**LEED 시스템(rating system)마다 같은 카테고리도 만점이 다름**. 예:

| 시스템 | 버전 | EA 만점 |
|--------|------|---------|
| BD+C (New Construction) | v4 | 33 |
| O+M (Existing Buildings) | v4 | **56** |
| ID+C (Commercial Interiors) | v4 | 38 |

하드코딩된 `VERSION_BD_C_MAX` 는 BD+C 기준만 가지고 있어서, O+M 건물을 BD+C 만점으로 나누면 달성률이 과대계산됨. PDF 스코어카드에는 실제 시스템별 만점이 `possible` 필드로 적혀있으므로 **이걸 가장 신뢰**.

### 5.3 각 단계의 실제 동작

**1순위 — PDF possible**:
```python
pdf_possible = cats_possible.get(cat, 0)     # PDF 에서 파싱된 값
if pdf_possible > 0:
    return float(pdf_possible)
```

**2순위 — rubric xlsx**:
```python
# data/rubrics/v4/BD+C_NewConstruction.xlsx 같은 공식 LEED 스코어카드
rubric_max = get_rubric_max(_RUBRIC_CACHE, version, leed_system, cat)
if rubric_max > 0:
    return float(rubric_max)
```
모듈 로드 시 `load_all_rubrics()` 가 `data/rubrics/` 아래 모든 xlsx 를 스캔해서 캐시함. 루브릭 파일이 없으면 이 단계는 자동으로 건너뜀.

**3순위 — 하드코딩**:
```python
# VERSION_BD_C_MAX[version][cat] 기본값 (BD+C 기준)
return float(VERSION_BD_C_MAX.get(version, VERSION_BD_C_MAX["v4"]).get(cat, 1))
```

이 fallback 덕분에 루브릭 xlsx 가 없는 버전 / 시스템도 에러 없이 처리됨 (정확도는 떨어질 수 있음).

## 6. `mapping_rules.yaml` — 크레딧 단위 매핑

### 6.1 역할

4절의 비율 환산은 **카테고리 합계** 레벨 변환. 하지만 "v2.2 의 어떤 크레딧이 v5 의 어떤 크레딧에 대응되는가" 같은 **크레딧 단위 매핑**은 수식이 아니라 **규칙 테이블**로 해결. 이게 `mapping_rules.yaml`.

### 6.2 파일 구조

현재 **108개 규칙**이 담겨있고, 카테고리별 섹션으로 구분됨 (EA / WE / SS / LT / MR / EQ / IP / O+M 전용 / 폐지).

각 규칙은 다음 7개 필드:

```yaml
- source_credit: "optimize energy performance"    # 구버전 크레딧명 (소문자 부분 매칭)
  source_versions: [v2.2, v2009, v3, v4, v4.1]   # 이 규칙이 적용되는 버전
  source_category: EA                             # 구버전 카테고리
  target_v5_code: EAc3                            # v5 크레딧 코드
  target_v5_name: Enhanced Energy Efficiency      # v5 크레딧명
  target_v5_category: EA                          # v5 카테고리
  confidence: high                                # high / medium / low
```

### 6.3 매칭 알고리즘 (`_lookup_credit_rule`)

```
입력: PDF 크레딧명 "Credit: Optimize Energy Performance 18pt"
      원본 버전 "v4"

1. 전처리: 소문자화, "Credit:" 등 접두사 제거
   → "optimize energy performance 18pt"

```

## 7. 수학 검증 (Hallucination Checker)

### 7.1 왜 "hallucination check"인가?

LLM 할루시네이션 체크가 아니라 **Rule Mapper 가 내놓은 숫자가 수식적으로 말이 되는지** 수학 제약을 검사하는 단계. LLM 안 씀.

### 7.2 5가지 검증 항목

| # | 검증 항목 | 예시 FAIL |
|---|----------|----------|
| 1 | 카테고리 점수 ≤ v5 최대값 | EA = 35 (v5 EA max=33) → FAIL |
| 2 | 총점 일관성 (sum ≈ total, 오차 0.5 이내) | 합=61, reported=65 → FAIL |
| 3 | **달성률 드리프트 ≤ 20%** | 원본 60% → v5 85% (+25%) → FAIL |
| 4 | 음수 점수 없음 | LT = -2 → FAIL |
| 5 | v5 에 없는 카테고리 없음 | mapped 에 IN 남아있음 → FAIL |

5개 전부 통과해야 PASS. 하나라도 실패 시 LLM Mapper 로 구제 경로 진입.

### 7.3 드리프트(Drift)란 무엇인가 — 핵심 지표

**정의**:
```
원본 달성률 = total_score_raw / version_total_max
v5 달성률   = total_score_v5  / v5_total_max(보통 100)
drift      = |원본 달성률 - v5 달성률|
```

**예시**:
- 원본(v2.2): 55점 / 69점 = **79.7%**
- 환산(v5): 75점 / 100점 = **75.0%**
- drift = |79.7% - 75.0%| = **4.7%** → 20% 이내, PASS

**의미**: "버전 환산이 이 건물의 실제 실력을 얼마나 왜곡했는가".
- drift = 0% → 완벽 보존
- drift 증가 → 환산 과정에서 점수 과대/과소 평가가 발생
- drift > 20% → 신뢰할 수 없음, LLM 구제 필요

### 7.4 왜 20% 임계치인가?

1. **LEED 등급 간격 폭과 비교**: Certified(40~49) vs Silver(50~59) 간격이 10점. 달성률로는 10%. 20% 드리프트는 **등급 2단계 이동** 가능 = 치명적 왜곡.
2. **v2009 크레딧 상세 부재 114건** 은 카테고리 비례배분으로 추정해야 해서 drift 가 구조적으로 발생 → 완전 무결 요구 시 전부 탈락.
3. **실증 tuning**: 처음엔 10%로 시작했으나 너무 엄격해서 LLM 호출 비용 폭증 → 20% 로 완화.

### 7.5 Drift 가 커지는 실제 원인

1. **SS→LT 교통 분리 비율 추정 오류** (가장 흔함): 크레딧 상세 없는 v2009 에서 교통 비율을 잘못 추정
2. **PDF 파싱 오류**: possible 값 추출 실패 → old_max 오판
3. **시스템 오매칭**: O+M 건물을 BD+C 만점으로 분모 계산
4. **버전 미식별**: `version="unknown"` → v4 fallback 적용되어 실제와 다른 분모 사용

## 8. 결과 통계 — 숫자가 나오는 방식

### 8.1 최종 수치

| 항목 | 값 | 의미 |
|------|-----|------|
| 처리 성공 | **460 / 460 (100%)** | 모든 PDF 가 `final_v5_data` 생성에 성공 |
| Rule 경로 | **413 건 (89.8%)** | math PASS → rule 점수 채택 |
| LLM 구제 경로 | **47 건 (10.2%)** | math FAIL → LLM 재매핑 점수 채택 |
| v5 평균 점수 | 45.1 / 100 | |
| 평균 drift | 11.9% | 전체 460건의 drift 평균 |

### 8.2 89.8% / 10.2% 가 나온 로직

`run_batch_standardization()` 이 460개 PDF 를 순회하며 각각:
```
pdf_ingest → csv_match → rule_mapper → hallucination_checker
                                           ↓
                                  math PASS → rule 점수 확정 (track="rule")
                                  math FAIL → LLM Mapper (track="llm")
```

최종 `standardization_track` 필드로 집계:
- `track == "rule"` 인 행의 개수 / 전체 460 = **89.8%**
- `track == "llm"` 인 행의 개수 / 전체 460 = **10.2%**

### 8.3 LLM 구제 47건의 특징

- 원본 버전이 **v2.2** 이거나 **credit 상세 없는 v2009** 에 집중
- 고급 소매 건물 (Prada / Tiffany / Gucci / Cartier 등) 에서 drift 높음 → 설비 투자는 큼(EA 높음)인데 SS/LT 제약 낮은 패턴
- 평균 drift = 28% (20% 임계 초과 조건 자체)

### 9.2 `ratio_{cat}` 이 핵심 ML feature 인 이유

```
ratio_{cat} = score_v5_{cat} / v5_max[cat]
            = (awarded / old_max × v5_max) / v5_max
            = awarded / old_max
            = "이 카테고리에서 원본 기준 몇 % 달성했나"
```

**버전이 달라도 동일한 의미** (달성률) 를 가지므로 ML 모델에 직접 투입 가능.
