# 02. LangGraph 파이프라인 설계

## 왜 LangGraph?

각 단계가 독립 노드로 분리 + 조건부 엣지로 분기 정책 집중 + State로 중간 결과 추적.

**대안과 비교**:

| 방식 | 장점 | 단점 |
|------|------|------|
| 단순 스크립트 | 간단 | 중간 결과 추적 어려움 |
| Rule만 | 재현성 100% | 의미 검증 불가 |
| LLM만 | 유연함 | 매번 결과 다름, 비용 ↑ |
| **LangGraph + Rule + LLM (이 연구)** | **재현성 + 의미 검증** | 구현 복잡 |

## 파이프라인 구조

### 노드 7개

1. **pdf_ingest**: PDF 파싱
2. **csv_match**: 프로젝트 디렉토리 매칭
3. **rule_mapper**: 결정론적 v5 매핑
4. **hallucination_checker**: 수학 검증 (LLM 없음)
5. **llm_validator**: LLM 의미 검증 (토큰 소모)
6. **llm_mapper**: LLM 재매핑 (구제 경로, 토큰 소모)
7. **finalize**: 최종 저장

### 전체 흐름

```
PDF Ingest → CSV Match → Rule Mapper → Hallucination Check
                                             ↓
           math PASS ──→ LLM Validator(target=rule)
                              ↓
                         Finalize (항상, Option A)

           math FAIL ──→ LLM Mapper → LLM Validator(target=llm) → Finalize
```

## Option A 원칙

**LLM은 점수를 바꾸지 않는다. 의견만 메타데이터로 기록한다.**

| 필드 | 의미 |
|------|------|
| `standardization_track` | 최종 경로 (rule/llm) |
| `llm_review_target` | LLM이 검증한 대상 (rule/llm) |
| `llm_review_is_valid` | LLM 판단: True/False |
| `llm_review_score` | 신뢰도 (0~1) |
| `llm_review_issues` | 지적 사항 |
| `llm_review_feedback` | 전체 의견 |

### 왜 Option A?

1. **재현성**: 점수는 항상 Rule이라 동일 입력 → 동일 출력
2. **논문 본론 유지**: "LLM 매핑 정확도" 증명할 필요 없어서 NLP 논문화 방지
3. **책임소재 명확**: 계산=Rule, 검증=LLM 역할 분리

## 핵심 코드

### 라우팅 (`graph.py`)

```python
def route_after_hallucination_check(state):
    if not OPENAI_API_KEY:
        return "finalize"           # API 없으면 Rule 결과 바로 저장
    if math_passed:
        return "llm_validator"      # V2: Rule도 LLM 검증
    return "llm_mapper"             # math FAIL: LLM이 재매핑

def route_after_llm_validation(state):
    return "finalize"               # Option A: 항상 저장
```

### 검증 프롬프트 분기 (`nodes.py llm_validator_node`)

**Rule 검증 프롬프트** (target="rule"):
```
"이 Rule 매핑 의미적으로 타당한가?
1. LEED 버전 특성 반영 (SS→LT 분리 등)
2. 크레딧 누락 없나
3. v5 신규 카테고리 배분 적절한가
4. 원본 등급과 v5 환산 점수 합리적인가"
```

**LLM 검증 프롬프트** (target="llm"):
```
"LLM 재매핑 결과의 할루시네이션/수치 오류 점검.
1. 카테고리 최대값 초과 여부
2. 등급 일관성
3. 달성률 드리프트 ≤ 20%"
```

## Rate Limit 대응

- `tenacity` exponential backoff (최대 5회, 2→32초)
- 호출 간 `time.sleep(2)` (TPM 30k 기준)
- 최종 실패 시 Rule fallback
