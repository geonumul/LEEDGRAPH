# 04. 심사 예상 질문 대응

## Q1. 왜 LangGraph 써요? 함수로 해도 되잖아요

**답변**:
1. 각 단계(PDF 파싱, 매핑, 검증)가 **독립 노드로 분리**돼 디버깅 쉬움
2. "어디서 LLM 불러올지"가 **조건부 엣지 한 줄에 집중** → 정책 변경 용이
3. State 객체로 중간 결과 다 추적 가능
4. 체크포인트로 8시간 실행 중 끊겨도 **resume 가능**

## Q2. 왜 Rule 먼저고 LLM 나중?

**답변** 3가지:
1. **재현성**: Rule = LEED 공식 루브릭 수식. 동일 입력 = 동일 결과. LLM은 매번 조금씩 다름.
2. **비용**: LLM 주 계산자로 쓰면 460건 × 반복 = 수백 달러. Rule은 API 0회.
3. **책임소재**: 계산=Rule, 검증=LLM 분리하면 어떤 숫자가 어디서 왔는지 명확.

## Q3. LLM이 Rule을 90% 거부했는데 Rule이 틀린 거 아니냐?

**답변**:
- threshold 0.8이 **보수적**이어서 거부율 높음 (75건 실험 기준)
- 실제로 LLM이 **명시적으로 승인한 7건**은 구체적 도메인 근거 제시:
  - "LT/SS 분리 올바름, IN/RP 폐지 정확히 반영"
  - "Gold(61점) → v5(46.94) 합리적"
- Rule이 틀렸다기보다 **LLM이 보완 지표** 역할
- Option A 채택으로 **점수는 Rule이 결정**, LLM은 uncertainty signal만 제공

## Q4. 460개로 일반화 되나?

**답변**:
- USGBC Public LEED Project Directory 기준 한국 **전수**
- 표본 추출이 아니라 **전수 조사**이므로 한국 시장 대표성 확보
- 해외 확장 시 LLM 프롬프트의 국가 맥락만 교체하면 재사용 가능

## Q5. Train 정확도 1.0은 과적합 아닌가?

**답변**:
- 맞음. Train 1.0 / CV 0.8174로 **gap 18%는 과적합 확실**
- 그래서 논문에는 **CV 값만 사용** (Train은 참고용)
- Holdout test set은 미도입 — **후속 과제**로 명시
- Robustness 분석으로 보완 (subset별 일관성 확인)

## Q6. v2009 건물 114건은 왜 크레딧 분석에서 빠졌나?

**답변**:
- v2009 PDF 포맷이 **카테고리 합계만 제공**, 크레딧 상세 정보 없음
- PDF 파서 한계
- `category_proportional` 방식으로 **카테고리 레벨** 분석에는 포함
- 크레딧 레벨 SHAP에서만 제외

## Q7. SS→LT 분리 비율의 출처는?

**답변**:
- v2.2/v2009의 SS에 교통 크레딧 약 7점 포함 (LEED 공식 루브릭 참조)
- `rule_mapper_node`에서 비율 기반 추정
- 하드코딩된 값이라 **Peer review 필요** — 한계점으로 명시

## Q8. LLM 리뷰 75건이 전체를 대표하는가?

**답변**:
- 16.3% (75/460) 커버리지
- 버전별로 고르게 분포하진 않음 (v4가 53건으로 편중)
- 논문에는 **"표본 리뷰"**로 포지셔닝
- 후속 과제: threshold 완화 + 전수 리뷰

## Q9. 매핑 규칙 107개는 누가 검증했나?

**답변**:
- LEED 공식 루브릭(v4 BD+C, v4.1, v5) 기반으로 연구자가 수동 작성
- **Peer review 미실시** — 한계로 명시
- 대안: credit_rule_hit_rate로 **규칙 커버리지 정량화**
  - v4 건물 평균 88%, v2009 평균 64%

## Q10. CV fold별 분산이 큰 이유는?

**답변**:
- fold별 76~89% (std 0.0502)
- 원인 추정:
  1. 각 fold의 구버전(v2.x) 표본 수 차이
  2. 건물 유형 다양성 (사무실/주거/상업/물류)
  3. 460건이 복잡한 feature space 대비 작음
- 향후: 층화 기준에 `original_version` 추가 검토

## Q11. 이 연구의 주요 기여는?

**답변** 4가지:
1. **버전 전수**: v2.0~v4.1 모두 포함 (기존은 1개 버전)
2. **LangGraph 아키텍처**: Rule + LLM 이중 검증의 재현 가능한 파이프라인
3. **한국 전수**: 460건 한국 시장 대표 데이터셋 공개
4. **SHAP 해석**: 크레딧 단위까지 상세 분석

## Q12. 후속 연구 방향은?

**답변**:
1. **Holdout test set 도입** 및 외부 검증
2. **LLM 리뷰 전수 확대** (threshold 완화로 비용 절감)
3. **추가 feature**: 건물 용도, 위치, 준공 연도, 설계사
4. **해외 데이터셋 확장**: 일본, 대만 등 아시아 비교
5. **구버전 특화 매핑**: v2.2 SS→LT 분리 비율 정교화
